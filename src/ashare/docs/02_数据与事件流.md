# V6 数据与事件流

## 一、数据源分层

### 1. 基础结构化数据
优先使用 Tushare 解决：

- 交易日历
- 股票列表与状态
- 日线行情
- 复权因子
- daily_basic
- 指数成分
- 财务发布日期索引
- 重要公告索引类字段（若可用）

### 2. 原始文本事件
与 Tushare 分开管理：

- 上市公司公告正文
- 新闻正文
- 政策新闻
- 行业事件

原因很简单：  
结构化数据和文本事件是两种完全不同的资产，不要混在一张表里。

## 二、事件湖分层

### Raw
只保证“抓到了”，不做判断。

字段示例：

- `raw_id`
- `source_type`
- `source_name`
- `publish_time`
- `crawl_time`
- `title`
- `content`
- `url`
- `security_code_hint`
- `company_name_hint`
- `industry_hint`

### Staging
做轻度清洗：

- 编码统一
- 去 HTML
- 去广告、去免责声明
- 标题标准化
- 正文分段
- 初步去重

### Curated / Event Store
完成抽取后进入正式事件表。

## 三、第一批事件类型（建议固定 12 类）

1. 业绩预告
2. 业绩快报
3. 正式财报
4. 回购 / 分红
5. 定增 / 融资
6. 重大合同 / 大单
7. 并购重组 / 资产出售
8. 高管增持 / 减持
9. 监管处罚 / 问询
10. 停复牌 / ST / 风险警示
11. 诉讼仲裁
12. 政策与行业重大事件

## 四、数据缺口引擎的三档刷新机制

### 档 1：硬性必更
每天都做：

- 交易日历
- 股票状态
- 日线基础表
- 原始事件流抓取

### 档 2：事件触发刷新
当某类事件出现时，重算相应特征：

- 业绩预告爆发 -> 重算 surprise 类特征
- 政策密集 -> 重算政策敏感度类特征
- 高频风险公告 -> 重算风险密度类特征

### 档 3：研究驱动新增
由研究脑决定是否纳入研究空间：

- 公告情绪强度因子
- 事件扩散强度因子
- 行业政策脉冲因子
- 风险公告拥挤因子
- 事件后波动分位因子

## 五、数据清单制度

建议维护 `data_inventory.json`，最少记录：

- `dataset_name`
- `path`
- `grain`
- `keys`
- `last_refresh_time`
- `expected_refresh_frequency`
- `freshness_status`
- `missing_ratio`
- `owner_module`
- `recompute_triggers`

## 六、事件到特征的映射机制

例如：

### 业绩预告
- `earnings_preannounce_direction`
- `earnings_preannounce_surprise_proxy`
- `days_since_preannounce`
- `earnings_event_density_20d`

### 回购 / 分红
- `buyback_flag_20d`
- `dividend_signal_strength`
- `capital_return_event_count_60d`

### 处罚 / 问询
- `regulatory_risk_event_count_20d`
- `regulatory_risk_intensity_60d`
- `days_since_last_regulatory_event`

### 政策事件
- `policy_sensitive_industry_pulse`
- `policy_event_density_industry_20d`
- `policy_positive_exposure_score`

## 七、事件到研究动作的映射机制

研究脑在看到事件后，不是直接下结论，而是先决定动作类别：

- `refresh_existing_dataset`
- `recompute_feature`
- `add_candidate_feature`
- `adjust_label_horizon`
- `adjust_universe_filter`
- `adjust_portfolio_constraints`
- `pause_strategy_branch`
- `prioritize_new_experiments`
