系统日常使用说明（AshareC# 混合架构版）

一、先用一句话理解当前架构

当前不是 “C# 已经替代 Python 运行时”。
当前真实结构是：

1. Python 仍然负责正式研究、release、执行、trade clock。
2. `launch_canonical.py` 仍是正式入口。
3. `main_research_runner.py` 仍是业务根入口。
4. `trade_clock_service.py` 仍是自动调度主进程。
5. `csharp_runtime_skeleton` 目前只是治理 / 编排 / 边界建模骨架，不直接替代研究与下单。

三、当前真正推荐的使用方式

日常正式链路不要理解成 “一个脚本一次跑完全部事情”。
当前稳定操作方式是分层：

1. 研究层
   `research_only`
   负责跑研究，生成组合建议，并发布对应 release。

2. 发布层
   `release_only`
   负责把现有组合建议重新冻结成正式 release。

3. 执行层
   `execution_only`
   负责读取 release，经过时间门禁和 safety 后进入 OMS / bridge。

4. 调度层
   `trade_clock_service.py`
   负责按时间触发 research / release / preopen_gate / simulation / midday_review / afternoon_execution / summary。


四、你平时最常用的命令

1. 跑轻量预检


2. 跑一次轻量联调

3. 跑正式研究链

5. 只做执行门禁检查


6. 盘中正式执行


7. 午间复盘


8. 下午二次执行


9. 手动启动调度器

```powershell

```

10. 查看调度器状态

```powershell

```


五、当前 profile 怎么选

1. `quick_test`
   最轻量的真实集成联调档。
   不是秒级 smoke test。

2. `daily_production`
   当前日常自动主链默认档。

   适合你明确接受更长运行时间时使用。

八、当前关于 shadow 的真实规则

这个部分很重要。

1. shadow 不是坏了，而是默认故意不自动运行。

2. 过去日报里会把旧 shadow OMS 快照误当成当天结果。
   这个汇总问题已经修正；现在如果当天 shadow 没跑，日报会明确把它标成不可用 / 未运行，而不是继续拿旧快照装成新结果。

3. 在没有 broker no-submit 隔离前，不建议把 shadow 当成正式盘中并行对照线。


九、当前关于 simulation gap 的理解

当前最该盯的不是短期收益，而是 simulation 的 desired vs actual gap。

新的自动化日报会额外给出 gap 的主因归类，优先识别：

1. `turnover_budget_truncation`
   组合想动，但被日内换手 / 预算约束截断。

2. `oms_continuity_carryover`
   上一轮未完成意图、延续单、carry/reconcile 还没收口。

3. `execution_fill_friction`
   有派单，但成交跟不上。

4. `residual_portfolio_gap`
   仍有剩余目标偏差，但前三类不显著。

十一、最后的操作原则

1. 正式运行优先走 `launch_canonical.py`。
2. `integrated_supervisor` 主要用于联调，不是正式精确交易主路径。
3. 正式精确交易优先走：
   `research_only -> release_only -> execution_only`
4. 不要把 `quick_test` 当成秒级 smoke test。
5. 不要把 shadow 当前状态理解成“自动并行对照线已经可用”。
